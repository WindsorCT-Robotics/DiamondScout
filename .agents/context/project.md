# Project Context

## Overview

This project is a flexible scouting application for the FIRST Robotics Competition. It is designed to be usable for any
game, allowing for custom scouting parameters, infraction definitions, game object scoring, pit scouting, and data 
analysis and aggregation based on your custom scouting parameters.

## Stack

Both Frontend and Backend will use the same version of .NET. They should be using the latest .NET Version, which is currently 10.

This project does not need to be restricted to only LTS versions of .NET.

### Frontend

The frontend stack is a Razor Class Library (RCL) that defines the UI components and behaviors of the application.
This RCL is designed to be used by both a MAUI Hybrid application that will be deployed to Android and iOS, and a
Blazor WebAssembly application that will be deployed to the web. An F# shim library will be used to interface with the
API, which will use Fable.Remoting to communicate with the backend. This shim library's functions will be used by the
RCL to implement component behaviors.

### Backend

The backend is an F# ASP.NET Core API that will use Fable.Remoting to communicate with the frontend. It will be split
into three projects: Common (the Domain Layer), Application (the Application Layer), and API (the public API).

### Persistence

The persistence layer will be implemented using KurrentDB (formerly known as EventStoreDB). Events will also be used to
construct a PostgreSQL database in a separate project.

## Architecture

### Domain

The domain layer will be implemented using F# in the `Common` project. It will be used to define the domain model and
follow domain-driven design principles using event sourcing.

Domain services shall include an event store interface. Said interface must be able to express persisting events and
running projections.

The domain layer shall also include the Fable.Remoting Interface records (which are not .NET Interfaces, but F# Records of functions).

All core business logic, business types, and value objects shall be defined in the domain layer.

All domain types shall be immutable.

Aggregates shall be defined using the `AggregateDefinition` type.
Events shall be generated by running Commands through the Decide function.
Events shall only be able to be generated or instanced in this method.

Evolve functions must *never* throw; all validation must be done in the Decide function and ***must be comprehensive***.

Identifier types should not be contained inside the entities they identify; they should be used to identify event streams.
They are excluded from the entity definition as they are not invariants; only invariant properties should be defined in an entity.

### Persistence

The persistence layer will be defined by two projects. One shall be an Event Store; the other a traditional SQL database.

The event store will be implemented using KurrentDB (formerly known as EventStoreDB).

Events will also be used to construct a database. An event handler specifically will be used to update the database
using Entity Framework Core. This allows for the state of the database to be reconstructed from the event store
if necessary. The database will be a PostgreSQL database.
